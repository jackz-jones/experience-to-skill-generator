# 早期版本局限性分析

> [English](LIMITATIONS.md)

本文档记录 `Experience-to-Skill Generator` 早期版本（纯规则引擎）的设计弊端与局限性，作为后续迭代的警示参考。

---

## 1. 核心问题：没有 AI 推理能力

当前脚本 `universal_skill_generator.py` 的整个分析链路是 **100% 纯硬编码规则**，没有接入任何 LLM / AI 模型。

**实锤证据：**

- 全部 import 均为 Python 标准库（`re`、`json`、`os`、`pathlib` 等），无 `requests`、`openai`、`anthropic` 等任何外部依赖。
- 无任何 HTTP 请求、API 调用、模型推理代码。
- 所有"分析"本质上都是正则表达式匹配 + 字符串操作。

---

## 2. 各核心函数的局限性

### 2.1 任务提取 `extract_tasks`

```python
# 实际逻辑：匹配含特定中文关键词的句子
if any(marker in line for marker in ["请", "需要", "帮我", "实现", "修复", "分析", "生成", "如何", "怎么"]):
    tasks.append(line[:160])
```

| 问题 | 说明 |
| --- | --- |
| 关键词覆盖面极窄 | 仅 7-8 个中文关键词，英文会话几乎全部漏掉 |
| 无语义理解 | "请问今天天气如何" 也会被当作任务提取 |
| 非指令式对话失效 | 讨论式、探索式对话中的隐含任务完全无法识别 |

### 2.2 步骤提取 `extract_steps`

```python
# 实际逻辑：正则匹配编号列表 + 特定中文开头词
pattern = re.compile(r"(?:^|\n)\s*(?:\d+[.、)]|-|\*)\s*([^\n]{8,180})")
# + 匹配 "首先/然后/接着/最后/验证/检查/运行" 开头的句子
```

| 问题 | 说明 |
| --- | --- |
| 只能抓格式化步骤 | 必须是 `1. xxx` 编号列表或特定开头词 |
| 自然语言描述的步骤全丢 | "先把配置改一下，然后重启服务" 这种口语化表述无法提取 |
| 无法理解步骤间的逻辑关系 | 提取的是孤立句子，不知道先后依赖 |

### 2.3 关键词提取 `extract_keywords`

```python
# 实际逻辑：正则提取 + 词频统计
candidates = re.findall(r"[A-Za-z][A-Za-z0-9_\-]{2,}|[\u4e00-\u9fff]{2,}", text)
```

| 问题 | 说明 |
| --- | --- |
| 纯词频统计，无语义权重 | 高频但无意义的词可能排在核心概念前面 |
| 停用词表极小 | 仅十几个停用词，过滤效果有限 |
| 无法识别专业术语和短语 | "机器学习" 会被拆成两个独立词 |

### 2.4 约束提取 `extract_constraints`

```python
# 实际逻辑：匹配含特定中文关键词的句子
if any(marker in clean for marker in ["不要", "必须", "禁止", "注意", "约束", "只能", "避免", "请勿"]):
```

| 问题 | 说明 |
| --- | --- |
| 仅 8 个中文关键词 | 英文约束（"must not"、"do not"、"avoid"）全部漏掉 |
| 无法理解隐含约束 | "这个接口有并发限制" 不含任何关键词，但确实是约束 |
| 误提取 | "注意这个功能很好用" 也会被当作约束 |

### 2.5 置信度计算 `calculate_confidence`

```python
# 实际逻辑：简单加分制
score = 0.2
if len(messages) >= 4:  score += 0.2
if tasks:               score += 0.2
if len(steps) >= 3:     score += 0.2
if any(role == "assistant"): score += 0.1
if any(role == "user"):      score += 0.1
```

| 问题 | 说明 |
| --- | --- |
| 跟内容质量完全无关 | 只看"有没有"，不看"好不好" |
| 容易虚高 | 4 条无意义消息 + 3 个误提取步骤 = 0.8 高置信度 |
| 无法反映 skill 的实际可用性 | 置信度高不代表生成的 skill 真的能用 |

## 3. 整体架构弊端

### 3.1 One-Shot 生成，无自检机制

```
输入 session → 正则提取 → 模板填充 → 输出 SKILL.md（结束）
```

- 没有任何"生成后检查"环节。
- 不会验证生成的 skill 是否准确覆盖了会话要点。
- 不会检测提取结果是否合理（比如任务和步骤是否匹配）。

### 3.2 全量分析但非增量生成

- 每次运行都重新分析所有 session 文件。
- 生成的 skill 与上次生成的完全独立，没有增量更新能力。
- 冲突策略（如 `rename`）仅避免文件覆盖，不实现内容层面的增量合并。

### 3.3 语言偏向严重

- 所有提取规则都以中文关键词为主。
- 英文会话的提取效果极差，几乎等于随机。
- 没有多语言适配机制。

---

## 4. 本质定位偏差

| 期望定位 | 实际定位 |
| --- | --- |
| 智能分析器：理解任意会话语义，准确提炼可复用经验 | 模板填充器：用正则抠出关键句子，塞进固定模板 |
| 适用于任意 input | 仅适用于中文、格式良好、包含明确指令词和编号步骤的对话 |
| 生成高质量可直接复用的 skill | 生成需要大量人工审核和修改的 skill 草稿 |

---

## 5. 改进方向

### 5.1 接入 LLM 做语义分析

用 AI 模型替代硬编码正则，实现真正的语义理解：

```
会话文本 → LLM 分析（理解意图/提取任务/识别步骤/发现约束） → 结构化输出
```

### 5.2 引入 ReAct 自检循环

生成后让 AI 自检，不满意则迭代优化：

```
生成 SKILL 初稿 → AI 自检（是否准确覆盖要点？） → 不满意 → 调整重新生成 → 满意 → 输出
```

### 5.3 增量分析能力

- 记录已分析过的 session 文件哈希。
- 仅对新增/变更的 session 做增量分析。
- 支持将新分析结果智能合并到已有 skill 中。

### 5.4 多语言支持

- 提取规则不应硬编码特定语言的关键词。
- 由 LLM 自动适配输入语言。
- 输出语言可配置。

---

## 6. 总结

早期版本的核心教训：

> **不要用硬编码规则去做需要语义理解的事情。** 正则表达式能处理的是"格式"，而不是"含义"。当输入是自然语言时，硬编码规则的覆盖面永远是有限的，且维护成本会随着规则数量指数增长。

这个版本作为 MVP 验证了"会话 → skill"这条路径的可行性，但要真正实用，必须引入 AI 推理能力。
